## 1. 테스트 계층 및 정의 (Practical Test Pyramid)
에이전트는 모든 테스트 작성 시 [실용적인 테스트 피라미드](https://martinfowler.com/articles/practical-test-pyramid.html) 원칙을 준수한다.

| 계층 | 정의 | 주요 검증 대상 | 도구 |
| :--- | :--- | :--- | :--- |
| **Unit Test** | 개별 단위 코드(함수, 메서드) 검증 | 비즈니스 로직, 유틸리티 코드 | JUnit5, Mockito, AssertJ |
| **Integration Test** | 모듈 간 상호작용 검증 | DB 연동, API 호출, 외부 시스템 결합 | SpringBootTest, MockMvc |
| **E2E Test** | 사용자 관점의 전체 흐름 검증 | HTTP API 요청-응답 전체 프로세스 | RestAssured, MockMvc |

## 2. 테스트 코드 작성 원칙 및 목표
* **테스트 커버리지:** * 모든 **핵심 비즈니스 로직**에 대한 테스트 필수 작성.
    * 패키지별 설정된 테스트 커버리지 기준 준수 (**전체 70% 이상**).
    * 모든 API에 대한 **End-to-End Test** 작성 필수.
* **외부 의존성 격리:** * `Mockito` 및 `MockMvc`를 사용하여 외부 의존성(Repository, 외부 API 등)을 Mocking하여 독립적인 테스트를 작성한다.
* **테스트 이점 극대화:** * 테스트 가능한 코드를 작성함으로써 자연스러운 **모듈화 및 SRP(단일 책임 원칙)** 준수를 촉진한다.
    * 테스트 코드를 통해 새로운 팀원의 온보딩 속도를 높이고 코드의 동작 이해를 돕는다.

## 3. 코드 구현 가이드 (Strict Rules)
에이전트는 코딩 시 다음의 제약 사항을 반드시 지켜야 한다.

* **객체 설계:** * **Immutable 객체 지향**: Setter 사용 금지(DO 제외). 객체 상태 변경 시 빌더 패턴을 통해 새로운 객체를 생성한다.
    * **일급 컬렉션 사용**: 일반적인 `Map` 사용을 금지하고 전용 컬렉션 객체를 생성한다.
    * **계층별 객체 분리**: `DTO(Request/Response)` ↔ `Domain Object` ↔ `Data Object(DO)`를 명확히 분리한다. (ex: UserRequest → User → UserDO)
* **SQL 및 DB:**
    * **Join/Subquery 제한**: Subquery는 사유 입증 시에만 허용한다. Join은 사유 입증 시 최대 2개 테이블까지만 허용한다.
    * **Outer Join 금지**: 명확한 join 조건을 식별하지 않는 경우 사용을 금지한다.
    * **DB 의존성 제거**: Procedure, Function 등 DB 의존적인 오브젝트 사용을 금지한다.
* **의존성 관리:** * 패키지 의존성 방향: `Domain` → `Controller` → `Service` → `Repository`
    * 각 최상위 Domain 객체 간 직접적인 Dependency 참조를 금지한다. (Util/Model 객체는 예외)

## 4. 코드 리뷰 및 협업 규칙
에이전트가 코드 리뷰를 수행하거나 MR을 작성할 때 다음 철학을 반영한다.

1.  **비자아적 프로그래밍 (Egoless Programming):** 코드에서 개인의 자아와 감정을 배제하고 공동 책임을 강조한다.
2.  **보이스카우트 법칙 (Boy Scout Rule):** 작업 후 기존 코드보다 조금이라도 더 깨끗한 상태로 개선하여 커밋한다.
3.  **검증 우선순위:** **로직 검증(최우선)** > 가독성 > 컨벤션 > 최적화 순으로 검토한다.

## 5. MR(Merge Request) 및 프로세스 가이드
* **MR 단위:** Jira 티켓 단위로 나누어 생성하여 리뷰어의 부담을 최소화한다.
* **머지 조건:** 최소 1건 이상의 **Approve**가 필수적이다.
* **리뷰 기한:** 워킹데이 기준 **2일 이내** 리뷰 완료를 원칙으로 한다.
* **선머지(긴급):** 프로덕션 장애 등 긴급 상황 시 사전 협의 후 선머지가 가능하며, 24시간 내 사후 리뷰 및 수정을 완료한다.
